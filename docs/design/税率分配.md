# Meme 币与金库借贷（规则摘要）

> 代币：**BABURU**（`baburu`）  
> 金库合约：**BABURU KINKO**（`baburu kinko`）  
> 执行平台：**BSC**

---

## 1. 基础信息

- 本币为平台发行 meme 币。
- 总量：**1,000,000,000**（固定上限）。

## 2. 税率与税金流向

| 方向 | 税率 |
| ---- | ---- |
| 买入 | 2% |
| 卖出 | 4% |

- 税收先以本币收取，再换为 BNB。
- 换得 BNB **100% 进入金库合约**。
- 金库仅被动接收 BNB（如 `transfer`），不存在主动“提税”流程。

## 3. 术语与订单生命周期

| 项 | 定义 |
|----|------|
| **借款** | 用户质押本币，从金库借出 BNB。 |
| **还款** | 归还该订单在借款时确定的等额 BNB；若存在罚金，则直接从该笔质押本币中扣除后再解押剩余部分。 |
| **T0** | 该笔订单借款成功的链上时间戳。 |

- 借款周期：6 天。
- 宽限期：3 天，即 **[6, 9) 天**，该区间允许还款但会从质押本币中扣除对应滞纳罚金。
- 超期：当订单到达 **>= 9 天**，该订单不可还款，进入清算销毁路径。

## 4. 时间与罚金（链上时间）

罚金按订单单独计算：  
`罚金 = 该订单质押本币数量 × 档位比例`  
罚金不再额外向用户钱包收取，而是**直接从该笔订单已质押的 BABURU 中扣除**；扣除部分统一通过 `transfer(0x...dead)` 处理，剩余质押返还给用户。

| 阶段 | 时间区间（自 T0） | 是否可还款 | 本币罚金（占质押本币） |
|------|-------------------|------------|------------------------|
| 提前还款 | [0, 1) 天 | 可 | 90% |
| 提前还款 | [1, 2) 天 | 可 | 60% |
| 提前还款 | [2, 3) 天 | 可 | 30% |
| 正常期内还款 | [3, 6) 天 | 可 | 0 |
| 宽限期（滞纳） | [6, 7) 天 | 可 | 30% |
| 宽限期（滞纳） | [7, 8) 天 | 可 | 60% |
| 宽限期（滞纳） | [8, 9) 天 | 可 | 90% |
| 超期 | >= 9 天 | 不可 | 质押本币 100% 销毁 |

> 实现提示：`>= 9 天`、`[6, 7)` 等边界比较应全合约统一（建议“左闭右开 + `>=`”）。
> 销毁口径统一：文档中“销毁”均指 `transfer(0x...dead)`，不使用 `burn()`；该处理不减少 `totalSupply`。

## 5. 可借 BNB 额度

在不超过金库可借池的前提下，单笔可借 BNB：

`可借 BNB = 本笔质押本币 / 可借份额分母 × (金库规模 BNB × ρ)`

**金库可借池（定义）**：本笔借款执行时，**实际借出 BNB 不得超过本笔执行前金库合约的 BNB 余额**（`address(this).balance`，同下项「金库规模 BNB」）。

其中：

- 金库规模 BNB：本笔借款执行前金库实时可用 BNB 规模（语义对齐 `address(this).balance`）。
- 可借份额分母（本笔借款时实时值，接受实时波动风险）：
  `可借份额分母 = 总量 − 非活跃本币量 − 活跃质押量`
- 总量口径：公式中的“总量”固定为本币**初始总量常量**（非实时总供应量）。
- 非活跃本币量（实时）：`黑名单地址实时持仓本币总量`（见 §8）。
- 活跃质押量（实时）：`当前金库合约内未结清订单的质押本币总量`。
- 口径约束：`非活跃本币量`统计时不包含“金库内活跃质押”部分，避免重复扣减。
- 风险约定：黑名单地址实时持仓波动带来的可借值变化由借款方自行承担。

补充规则：

- 分母若 **<= 0**，本笔借款必须失败（整笔 revert）。
- 默认 `ρ = 0.7`，可调范围与权限见 §7。
- 借款前，前端基于用户输入的本币数量给出参考可借值（`refBorrow`）。
- 用户可通过滑杆设置最小可借比例（`minBorrowBps`），默认 `9500`（95%），且 `1 <= minBorrowBps <= 10000`。
- 借款执行时，合约实时计算当前可借值（`currBorrow`），并校验：  
  `currBorrow * 10000 >= refBorrow * minBorrowBps`
- 若校验不满足，则回滚当前借款操作（整笔 revert）。
- 允许多笔借款，支持多订单一键处理（见 §6）。

## 6. 订单处理与查询

用户可在前端查询自己地址**当前未完成订单**，以时间倒序列表展示；订单信息至少包含：质押本币、借出 BNB、借款时间、预估到期时间、订单状态。

一键还款 / 批量处理约定：

- 批量处理时，用户勾选需处理订单；合约计算当前“勾选订单集合”应还 BNB，用户确认后提交。
- 其中 `>= 9 天` 待销毁订单前端不可选；即使被传入，合约也自动执行销毁且不计入应还 BNB 总额。
- 调用时可传入多笔订单。
- 逐笔判定规则：
  - 未超期订单：仅允许处理调用者本人订单，按对应档位完成还款，并从该笔已质押 BABURU 中扣除罚金后返还剩余部分。
  - 超期订单（`>= 9 天`）：触发该笔质押 100% 销毁。
- 批量处理中，超期订单不会阻塞其他未超期订单的还款处理。
- 若传入 BNB 与“传入订单集合”的应还总额不一致，则回滚。

便捷清算与只读统计：

- 合约可额外提供**全局整理金库**便捷入口：无需传入具体订单 ID，即可按给定数量上限遍历并处理当前所有 `>= 9 天` 的超期借款。
- 合约可额外提供**待清算汇总只读接口**：返回当前待清算借款数量与待清算质押本币总量，供前端概览卡与整理入口展示。

存储约束：

- 合约内只保存未超期订单和超期但未执行销毁的订单。
- 已完成订单不再保存相关信息。

## 7. 金库权限（信任边界）

在不违反「金库 BNB 不得向任意地址随意转出」的前提下，治理可拥有有限、可审计权限。权限模型应与实际实现一致（如 Owner / 多签 / Timelock / 可升级性），并接受审计。

> 实现提示：若存在 upgrade / proxy 或可长期暂停能力，需在白皮书与合约中单独披露，并与「无人可随意转走 BNB」并列说明权限边界。

### 7.1 原则

| 项 | 说明 |
|----|------|
| **BNB 不可随意转出** | 任何角色（含开发者）不得将金库内 BNB 任意划转、提现或转给任意地址。仅允许白名单路径：借款、还款回流、税金入账等。 |

### 7.2 公共接口（无特权）

| 项 | 说明 |
|----|------|
| **公共清算** | 对已满 9 天且应结清订单发起清算，推动质押销毁。**任意地址可调用**，无需 Owner/治理权限。 |
| **整理金库（便捷入口）** | 可选：提供面向前端的一键整理入口，用于按数量上限批量推进当前全部超期借款进入销毁路径；其语义属于公共清算的便捷封装。 |

### 7.3 特权接口（合约内受控；部署文档中可称「开发者预留」）

下列接口**非普通用户**可调用；链上一般以 `onlyOwner`、多签或 Timelock 实现。**具体角色名以合约为准**（Owner / 治理 / 多签与是否由开发者代持无关）。

| 项 | 说明 |
|----|------|
| **可调：可借比例 ρ** | 可调整 §5 中 ρ（当前 0.7）。建议链上设置上下界、Timelock、事件。 |
| **增删黑名单地址列表** | 用于动态计算可借份额分母（如 LP、交易所托管、约定销毁地址 `0x…dead`）。变更实时生效；并发借款以交易执行时链上状态为准。 |
| **启动 / 暂停** | 暂停仅影响「新借款开放状态」；不影响既有订单的还款与到期清算。 |
| **初始化与配置** | 本币合约地址与初始数量、税币流入合约地址、销毁地址等启动前配置；与 §9 启动流程一致。 |

## 8. 常规黑名单地址

1. LP 地址
2. 交易所托管地址
3. 销毁地址 `0x…dead`

## 9. 启动流程

1. 设置本币合约地址与初始数量。
2. 设置税币流入合约地址（该地址同时为借出币种托管地址）。
3. 设置销毁地址为 `0x...dead`，代币“销毁”统一为向该地址 `transfer()`。
4. 设置黑名单地址列表。
5. 调用启动接口。
